Fiori Fundamentals
Q1 What is SAP Fiori? Why was it introduced?
Answer
SAP Fiori is SAP's modern UX (User Experience) framework — role-based, responsive applications accessible from any device: desktop, tablet, or mobile.

Why it was introduced:

Old SAP GUI was complex, ugly, and not user-friendly at all
Businesses needed mobile-compatible interfaces for on-the-go use
Users wanted simple, task-based apps instead of cluttered screens
Business required faster access to key tasks without navigating deep menus
Think of Fiori like a smartphone home screen — each tile is a shortcut that opens one specific app for one specific job.
Q2 What are the main components of SAP Fiori architecture?
Answer
1
Frontend Server (FES) — Hosts Launchpad, Catalogs, Tiles, OData registration
2
SAP Gateway — Middle layer connecting Frontend to Backend via OData protocol
3
Backend Server (ECC / S/4HANA) — Business logic, authorization objects, real data
4
OData Services — The API channel carrying data between all layers
FLP (Fiori Launchpad) is the central homepage where users land after login. All tiles live here and every Fiori session starts here.
Q3 What is the difference between Catalog, Group, Space, and Page?
Answer
Catalog — Gives authorization to see tiles. Contains target mappings. Without catalog assigned in PFCG role, user cannot see or open the tile.
Group — Older method to display tiles on launchpad home page. Controls visibility only, not authorization. Being replaced by Spaces.
Space — New SAP recommended structure replacing Groups. Better organized, more flexible layout for S/4HANA systems.
Page — Lives inside a Space. Contains sections where tiles are arranged and displayed to the user.
Key rule: Catalog = Authorization to access. Space/Page = Visibility and layout only. You need BOTH for the tile to show and work correctly.
Q4 What are the different types of Fiori Apps?
Answer
Transactional AppsAnalytical AppsFact Sheet Apps
Transactional — For creating, editing, approving business records. Example: Create Purchase Order, Approve Leave Request
Analytical — For viewing charts, dashboards, KPIs, and reports. Example: Sales Overview Dashboard, Budget Analysis
Fact Sheet — Shows key contextual information about a business object. Example: Supplier Details, Material Info Page
Security Layers & Roles
Q5 What are the major security layers in SAP Fiori?
Answer

Fiori security has 4 distinct layers — every single layer must be correctly configured for the app to work:

1
Frontend Auth — Catalog & Tile authorization on the Frontend server
2
OData Auth — User permission to call the specific OData service
3
Gateway Auth — Gateway communication and service registration allowed
4
Backend Auth — Authorization objects checked in the backend system
If even ONE layer is missing or incorrect, the app will either not show, give HTTP 403, or throw a data loading error. Always check all 4 layers systematically.
Q6 What is the difference between ECC Security and Fiori Security?
Answer

ECC Security

  • Transaction-based access
  • Users access via T-Codes
  • PFCG roles only needed
  • SAP GUI (desktop app)
  • Simpler auth model

Fiori Security

  • App/tile-based access
  • Users access via Tiles
  • Catalog + OData + Backend
  • Browser / mobile
  • Multi-layer auth model
In Fiori, just adding a T-code to a backend role is NOT enough. You also need: Catalog assignment, OData service activation, and SICF node active. Missing any one = app won't work.
Q7 What role types are required for Fiori and what goes inside each?
Answer

You always need two separate roles for Fiori to work correctly:

Frontend Role

  • Business Catalog
  • Space / Page assignment
  • Target Mapping
  • OData service registration
  • Created in PFCG (FES)

Backend Role

  • T-Codes (transactions)
  • Authorization objects
  • OData authorization
  • Business data access
  • Created in PFCG (Backend)
Frontend role = what the user can SEE on the launchpad. Backend role = what the user can DO and ACCESS in the system.
Tiles, OData & Navigation
Q8 What is a Tile and what is Target Mapping?
Answer
Tile — Visual button/icon on the Fiori Launchpad. It is what the user sees and clicks to open an application. Can show live data (dynamic tiles) or be a static shortcut.
Target Mapping — The bridge between a tile and the actual application. It defines: which Semantic Object + Action → maps to which application URL/component to open.
Without a Target Mapping configured correctly, the tile will show on the launchpad but clicking it either does nothing or throws a "Cannot resolve navigation target" error.
Q9 What is a Semantic Object and Semantic Action? Give an example.
Answer
Semantic Object — A logical business entity name. Examples: Material, PurchaseOrder, Supplier, Invoice, SalesOrder, Employee
Semantic Action — The operation to perform on that object. Examples: display, create, manage, approve, edit, post
Semantic Object = PurchaseOrder
Action = display
Intent URL = #PurchaseOrder-display

Full URL example:
#SalesOrder-display?SalesOrder=4500001234
If the Semantic Object or Action in Target Mapping does not match what the tile expects, you get the "Cannot resolve navigation target" error.
Q10 What is OData? Why is it important in Fiori Security?
Answer

OData (Open Data Protocol) is the REST-based API/communication standard used by Fiori to exchange data between the frontend and SAP backend.

Fiori app requests data → the registered OData service is called
OData request passes through SAP Gateway → hits the Backend server
Backend processes, checks auth, returns data back through OData → displayed in app
If OData service is not activated OR user has no authorization to call it → the app will not load data, even if the tile is visible on launchpad.

Activate OData

  • /IWFND/MAINT_SERVICE

Test OData

  • /IWFND/GW_CLIENT
Q11 What is SICF and why is it important for Fiori?
Answer

SICF (Service Implementation using ICF) is the SAP transaction used to activate or deactivate HTTP/HTTPS services — which includes all Fiori app services.

Every Fiori app requires its SICF node to be in ACTIVE status to load in a browser
If SICF node is inactive → user gets HTTP 404 or blank page when opening the app
Path to activate:
SICF → sap → bc → ui5_ui5 → sap → [your_app]
Right-click → Activate Service
Transaction Codes Reference
Role & User Management
PFCG
Role Maintenance
Create, edit, generate, and assign roles to users. The most-used T-code in all of SAP Security — you'll use this every day.
SU01
User Maintenance
Create or modify user accounts, lock/unlock users, reset passwords, assign roles directly to a single user.
SU10
Mass User Maintenance
Assign roles, lock/unlock, or change validity for multiple users at once. Saves time for bulk operations.
SU01D
User Display (Read-Only)
View user details, assigned roles, and logon data without any edit permissions. Safe to share with non-admin staff.
SU21
Authorization Object Maintenance
Create and manage authorization objects and their fields. Used when creating custom authorization checks for custom apps.
Troubleshooting & Auth Trace
SU53
Authorization Error Check
Shows the last failed authorization check for any user. Always the FIRST place to check when a user reports an auth error. Run it right after the user hits the error.
STAUTHTRACE
Authorization Trace
Real-time trace of all authorization checks for a specific user. More powerful than SU53 — captures the full sequence of auth checks made when user performs an action.
ST01
System Trace
Full system-level trace including authorization, RFC calls, and more. Use when SU53 and STAUTHTRACE are not enough to pinpoint the exact issue.
SU56
User Buffer Analysis
Shows all authorization objects currently in a user's buffer. Use after assigning a new role to verify the new authorization values are actually loaded.
OData & Gateway
/IWFND/MAINT_SERVICE
OData Service Activation
Register and activate OData services on the Frontend/Gateway server. Required step every time a new Fiori app is set up. Check here first if "Failed to load metadata" appears.
/IWFND/GW_CLIENT
Gateway Test Client
Built-in browser-like tool to test OData service calls directly in SAP. Use it to confirm whether a service returns data correctly, before blaming auth or the app.
/IWFND/ERROR_LOG
Gateway Error Log
Error log on the Frontend/Gateway server side. Shows failed OData calls, timeouts, and service errors. Check here when the Gateway layer seems to be failing.
/IWBEP/ERROR_LOG
Backend OData Error Log
Error log on the Backend server for OData model and data provider errors. Check here when the issue is on the backend side of the OData service.
Fiori Launchpad & Catalog
/UI2/FLPD_CUST
Launchpad Designer (Cust.)
Create and manage catalogs, groups, tiles, and target mappings in the customer namespace. Main tool for Fiori administrators to set up the launchpad.
/UI2/FLPCM_CUST
Catalog Manager
Manage and assign catalogs to PFCG roles. Use this to link a catalog to a role or verify which roles have a specific catalog assigned.
/UI2/FLP
Open Fiori Launchpad
Launches the Fiori Launchpad directly from SAP GUI. Useful for testing the launchpad as-is without opening a browser separately.
/UI2/INVALIDATE_CLIENT_CACHES
Clear Frontend Cache
Clears cached launchpad data on the frontend server. Use this when tile or catalog changes are not visible even after role generation. Always run after making catalog changes.
/UI2/CHIP_SYNCHRONIZE_CACHE
Sync Tile/Chip Cache
Synchronizes the tile chip cache after catalog or tile configuration changes. Run alongside /UI2/INVALIDATE_CLIENT_CACHES when tiles are not updating.
Service Activation & Monitoring
SICF
ICF Service Activation
Activate or deactivate HTTP services for Fiori apps. Every Fiori app node must be ACTIVE here or the app gives HTTP 404. Check here when the app doesn't load at all.
SMICM
ICM Monitor
Monitor the Internet Communication Manager — checks HTTP/HTTPS connections, ports, and SSL. Use when network-level connectivity to the Fiori server seems broken.
SMGW
Gateway Monitor
Monitor SAP Gateway connections, performance, and active connections between frontend and backend systems.
SM21
System Log
View system-level logs including security events, runtime errors, and critical failures. Useful for HTTP 500 errors and backend crashes.
Real-World Troubleshooting Scenarios
Sc1 User can see the tile but gets an authorization error when clicking it. What will you check?
Step-by-Step Approach
1
Run SU53 for the affected user immediately after they hit the error — it shows the last failed authorization object and the values that were checked vs. what was allowed.
2
Enable STAUTHTRACE for that specific user, ask them to click the tile again, then stop the trace and analyze all failed auth checks. More complete than SU53.
3
Open PFCG → check the user's assigned backend roles → verify the missing authorization object is present with correct values (activity, org levels).
4
Check OData service authorization — the user's backend role needs a specific auth object to call the OData service. Verify it's present.
5
Add the missing auth object/values to the correct role → Save → Generate the role profile → ask user to log off and log back in → retest.
Important: Never add authorization objects directly to the user profile (SU01). Always add to a role first, then assign the role to the user. Direct profile assignment is a security violation.
Sc2 User cannot see the tile at all on the launchpad. What will you check?
Step-by-Step Approach
1
Check if the correct Catalog is assigned in the user's PFCG frontend role. Go to PFCG → Menu tab → verify catalog is there and role is generated (green traffic light).
2
For S/4HANA systems — verify Space and Page assignment is done correctly in the role. The tile must be in a Page that's inside an assigned Space.
3
Check if the user's role is properly generated in PFCG. If the authorization data (traffic light) shows yellow or red, regenerate the profile and re-assign.
4
Run /UI2/INVALIDATE_CLIENT_CACHES to clear the server-side launchpad cache. Then run /UI2/CHIP_SYNCHRONIZE_CACHE as well.
5
Ask user to clear their browser cache (Ctrl+Shift+Del), close browser completely, and log into Fiori fresh. Browser cache often holds old launchpad state.
Tile not visible = catalog issue in 90% of cases. Always start with the catalog assignment check in PFCG before investigating anything else.
Sc3 App shows "Failed to load metadata" error after clicking a tile. How do you fix it?
Step-by-Step Approach

This error means the app cannot load the OData metadata document — the blueprint it needs to understand the service structure.

Go to /IWFND/MAINT_SERVICE → check if the OData service is activated (green status)
Test the OData service using /IWFND/GW_CLIENT — call the $metadata endpoint and see if it returns data or an error
Verify the SICF node for the app is active — go to SICF and check sap/bc/ui5_ui5 path for the specific app
Check /IWFND/ERROR_LOG for Gateway-side errors — look for the specific service name and read the error detail
Check /IWBEP/ERROR_LOG for backend OData model errors — sometimes the backend data provider has a bug or missing config
Open browser Developer Tools → Network tab → reproduce the error → look for the failed HTTP request and read the server response body
Sc4 HTTP error codes in Fiori — what does each one mean and where do you look?
Answer
403
Forbidden — Authorization failure. User is authenticated but lacks permission. Check: SU53 → STAUTHTRACE → backend role auth objects → OData service authorization.
404
Not Found — Service or URL does not exist or is not activated. Check: SICF node status → /IWFND/MAINT_SERVICE → verify the app URL path is correct.
500
Internal Server Error — Backend program crashed or threw an exception. Check: SM21 system log → /IWBEP/ERROR_LOG → contact ABAP developer if it's a code error.
Nav
Cannot resolve navigation target — Semantic Object or Action mismatch. Check: Target Mapping in /UI2/FLPD_CUST → verify the Semantic Object + Action combination is correctly defined.
Advanced Topics (3+ Years Experience)
A1 Explain the complete end-to-end flow when a user clicks a Fiori Tile
Complete Flow
1
User opens browser and logs into Fiori Launchpad (FLP) with their credentials
2
Launchpad checks Catalog authorization in the user's frontend role → only tiles from assigned catalogs are rendered on the homepage
3
User clicks a tile → Launchpad reads the Target Mapping associated with that tile
4
Target Mapping resolves the Semantic Object + Action into an intent URL (e.g. #PurchaseOrder-display)
5
The Fiori app component is loaded → it makes an OData service call through the SAP Gateway
6
SAP Gateway validates the request and forwards it to the Backend server
7
Backend runs AUTHORITY-CHECK statements → checks auth objects for the user's backend role
8
If authorized, data is fetched from backend tables and returned via OData response
9
App renders the data in the browser → user sees the application fully loaded
This 9-step flow is the most commonly asked advanced question in Fiori Security interviews. Know every step and be ready to explain where each error type (403, 404, metadata) fits in this flow.
A2 Difference between Business Catalog and Technical Catalog
Answer

Business Catalog

  • Assigned directly to PFCG roles
  • Contains tiles visible to end users
  • Business-user facing
  • Aggregates tiles from technical catalogs
  • Naming: SAP_xxx or Z_BUSINESS_xxx

Technical Catalog

  • NOT assigned to roles directly
  • Source/library of tiles and target mappings
  • Admin/developer facing
  • Contains raw target mappings
  • Naming: SAP_TECH_xxx or Z_TECH_xxx
Think of it like this: Technical Catalog = the full library of books. Business Catalog = the curated reading list selected for a specific role or team, pointing to books in that library.
A3 What is Intent-Based Navigation (IBN) in Fiori?
Answer

Intent-Based Navigation (IBN) is Fiori's standard way for apps to navigate to each other using a generic URL pattern instead of hardcoded application IDs.

URL Pattern: #SemanticObject-action?param=value

Example 1: #PurchaseOrder-display
Example 2: #SalesOrder-manage?SalesOrder=5000001
Example 3: #Employee-overview?pernr=12345
Apps don't hardcode links to each other — they just declare an "intent" and the Launchpad resolves which app handles it
Gateway resolves the intent via Target Mapping → determines which app UI component to load
This makes all Fiori apps loosely coupled, reusable, and easy to replace without breaking navigation
A4 How do you secure custom Fiori applications?
Answer
1
Create a custom authorization object in SU21 with the required authorization fields (activity, org levels, etc.)
2
Add the AUTHORITY-CHECK statement in the backend ABAP code (OData data provider class) to enforce the check at runtime
3
Create a custom catalog in /UI2/FLPD_CUST → add a new tile and configure the target mapping with correct Semantic Object + Action
4
Register the custom OData service in /IWFND/MAINT_SERVICE so the Gateway knows about it
5
Activate the SICF node for the custom app under SICF → sap/bc/ui5_ui5 path
6
Create a PFCG frontend role with catalog + Space/Page, and a backend role with the custom auth object → assign both roles to users
Never skip the backend authorization check! Frontend-only security can be bypassed by directly calling the OData service URL. Always enforce security at the backend layer.
